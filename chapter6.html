6.1無線安全威脅概觀

無線網路相較於有線網路，在安全性上面臨一個根本性的挑戰：訊號在空中傳播，任何人只要在覆蓋範圍內都能接收到訊號，不需要實體接觸到線路。這使得無線網路天生暴露在竊聽、偽裝和注入攻擊之下。

無線網路的安全威脅可以從三個層面理解：

機密性威脅
攻擊者被動地竊聽（Eavesdropping）空中傳輸的資料。若資料未加密或加密強度不足，攻擊者可以重建明文內容，取得帳號密碼、通訊內容、敏感文件。
完整性威脅
攻擊者篡改（Tampering）傳輸中的資料訊框，或注入偽造的管理訊框（如偽造 Deauthentication 封包）強迫裝置斷線或連接惡意 AP。
可用性威脅
阻斷服務（DoS）攻擊，包含持續發送 Deauthentication 訊框（De-auth Flood）讓所有裝置斷線，或以持續無線電訊號干擾整個頻段（RF Jamming）。無線 DoS 幾乎無法完全防禦，只能減輕影響。

無線安全的核心三要素

無線安全三要素（CIA for Wireless）： ┌─────────────────────────────────────────────────────────┐ │ │ │ 認證（Authentication） │ │ → 確認「你是誰」：防止未授權裝置連接網路 │ │ → WPA-Personal（密碼）/ WPA-Enterprise（憑證/帳號） │ │ │ │ 加密（Encryption） │ │ → 確保「資料機密性」：防止空中訊號被竊聽解讀 │ │ → WEP（RC4，已破）→ TKIP → CCMP（AES）→ GCMP（AES） │ │ │ │ 完整性（Integrity） │ │ → 確認「資料未被篡改」：防止訊框被中途修改 │ │ → MIC（Message Integrity Code）保護資料正確性 │ │ → 802.11w（PMF）保護管理訊框不被偽造 │ │ │ └─────────────────────────────────────────────────────────┘

6.2WEP：已破解的先驅

WEP（Wired Equivalent Privacy，有線等效保密）是 1997 年 IEEE 802.11 標準最初定義的無線加密機制，設計目標是讓無線網路達到「與有線網路相當」的安全等級。這個目標從未被達到，WEP 已於 2004 年被 IEEE 正式棄用，但至今仍有老舊裝置在使用，是極大的安全隱患。

WEP 的技術設計

WEP 使用 RC4 串流加密演算法，以靜態預共享金鑰（40 位元或 104 位元）加上 24 位元的初始化向量（IV, Initialization Vector）組成金鑰串流，對資料進行加密。完整性驗證使用 CRC-32 校驗碼。

WEP 的致命弱點

IV 重複使用問題
24 位元的 IV 只有 16,777,216 種可能值。在繁忙的網路上，IV 在數小時內就會重複出現。攻擊者收集到足夠多使用相同 IV 的加密封包後，可以進行統計分析破解金鑰。
靜態金鑰設計
WEP 金鑰由全體用戶手動設定且長期不變。任何一個知道密碼的人離職後，密碼仍然有效（除非所有裝置都更換密碼），無法實現個人身分認證。
CRC-32 完整性可操縱
CRC-32 是設計用於偵測傳輸錯誤的校驗碼，不是加密雜湊函數，無法防止主動攻擊。攻擊者可以修改加密封包中的部分位元並重新計算有效的 CRC-32，繞過完整性驗證。
實際破解時間
使用 Aircrack-ng 等公開工具，在有足夠封包（約 40,000～85,000 個含 IV 的封包）的前提下，可在數分鐘內破解 WEP 64 位元金鑰，104 位元同樣在數十分鐘內可破。
WEP 已於 2004 年正式廢止

任何仍在使用 WEP 的網路應立即升級。若設備只支援 WEP，唯一安全的選擇是更換設備。使用 WEP 加密等同於完全沒有加密保護。

6.3WPA：過渡方案

面對 WEP 的嚴重弱點，Wi-Fi Alliance 在 2003 年推出 WPA（Wi-Fi Protected Access）作為過渡方案，在等待 IEEE 802.11i 標準完成的期間提供更強的安全保護。WPA 的設計刻意相容現有 WEP 硬體，可透過韌體更新升級，而無需更換設備。

WPA 的核心改進：TKIP

WPA 引入 TKIP（Temporal Key Integrity Protocol，暫時金鑰完整性協定）取代 WEP 的靜態 IV 機制：

Per-Packet Key Mixing
TKIP 為每個封包動態產生不同的加密金鑰，根本解決了 WEP 的 IV 重複問題。即使攻擊者擷取大量封包，也無法用統計方式破解。
MIC（Michael）
引入 MIC（Message Integrity Check）訊框完整性驗證，取代 CRC-32，防止攻擊者篡改加密封包。TKIP 的 MIC 演算法稱為「Michael」。
IV 序號計數器
TKIP 使用 48 位元的 IV 序號（TSC），並要求接收端拒絕序號重放的封包，防止重放攻擊（Replay Attack）。

WPA-TKIP 的後續弱點

儘管 TKIP 修補了 WEP 的主要弱點，其底層仍使用 RC4 加密演算法，且 MIC（Michael）演算法被發現存在設計弱點。2008 年研究人員展示了針對 TKIP 的部分破解攻擊（Beck-Tews 攻擊），2009 年更被進一步擴展（Ohigashi-Morii 攻擊）。TKIP 已於 2012 年被 IEEE 廢棄，不應繼續使用。

6.4WPA2：長期主流標準

WPA2 於 2004 年發布，完整實作 IEEE 802.11i 標準，以 CCMP（Counter Mode with CBC-MAC Protocol）取代 TKIP，採用 AES（Advanced Encryption Standard）區塊加密，提供遠比 RC4 強大的安全保護。WPA2 成為此後近 15 年的主流無線安全標準。

CCMP / AES 的技術優勢

AES-128 加密強度
AES 使用 128 位元金鑰，目前無已知可行的實際破解方法。即使以現有最快的電腦進行暴力破解，也需要比宇宙年齡還長的時間。
CCMP 完整性保護
CCMP 使用 CBC-MAC（Cipher Block Chaining Message Authentication Code）同時提供加密和完整性驗證，兩者綁定，攻擊者無法在不被偵測的前提下修改加密資料。
重放保護
CCMP 標頭包含 48 位元的封包序號，AP 和客戶端會拒絕重複序號的封包，防止重放攻擊。

WPA2 的四向握手（4-Way Handshake）

WPA2 的身分認證完成後（PSK 模式或 802.1X），AP 和客戶端透過四向握手協商用於本次連線的PTK（Pairwise Transient Key，成對暫時金鑰），每次連線都會產生全新的 PTK，確保每次連線的加密金鑰不同。

WPA2 四向握手（4-Way Handshake）： AP（Authenticator） Client（Supplicant） | | |──── 訊息 1：ANonce ──────────────→| AP 傳送隨機數 ANonce | | Client 產生 PTK = PRF(PMK, ANonce, SNonce, MACs) |←─── 訊息 2：SNonce + MIC ─────────| Client 傳送 SNonce，用 PTK 計算 MIC 驗證 | AP 用相同材料重建 PTK，驗證 MIC | |──── 訊息 3：GTK + MIC ───────────→| AP 傳送群組金鑰（GTK），MIC 驗證 |←─── 訊息 4：ACK ─────────────────| Client 確認收到 GTK | | [PTK 安裝完成，開始加密資料傳輸] PTK = Pairwise Transient Key（本次連線的單播加密金鑰） GTK = Group Temporal Key（廣播 / 多播流量的加密金鑰） PMK = Pairwise Master Key（由 PSK 或 802.1X 認證推導出）

WPA2 的已知弱點

KRACK（Key Reinstallation Attack）
2017 年揭露的 KRACK 攻擊針對四向握手的訊息 3，利用重放訊息 3 讓客戶端重新安裝已使用的金鑰，導致 CCMP 計數器歸零，進而解密資料。大多數廠商已於 2017 年底發布補丁修復。
PSK 離線字典攻擊
攻擊者擷取四向握手封包（公開工具即可做到），接著對 PMK 進行離線暴力破解（字典攻擊）——無需連接到目標網路。若 PSK 密碼夠短或包含在字典中，可能在數分鐘至數小時內被破解。這是 WPA2-Personal 最大的安全弱點。
PMKID 攻擊（2018）
研究人員發現不需擷取完整四向握手，只需從 AP 廣播的 EAPOL 訊框中提取 PMKID 值，同樣可以進行離線字典攻擊，且比 KRACK 更容易執行。
WPA2-Personal 密碼長度的重要性

針對 WPA2-Personal（PSK）的字典攻擊效率完全取決於密碼強度。一個 8 字元的常見單詞密碼可能在數分鐘內被 GPU 破解；而一個隨機 20 字元的密碼（包含大小寫、數字、符號）在目前的計算能力下幾乎不可能被暴力破解。請務必使用足夠長且隨機的 PSK。

6.5WPA3：現代安全基石

WPA3 於 2018 年由 Wi-Fi Alliance 發布，正面回應 WPA2 的所有已知弱點，帶來多項根本性的安全改進。Wi-Fi 6 認證設備必須強制支援 WPA3。

WPA3 的四大核心改進

SAE（Simultaneous Authentication of Equals）取代 PSK
SAE（俗稱 Dragonfly 握手）是 WPA3-Personal 的認證機制，根本解決了 WPA2 的離線字典攻擊問題：
（1）前向保密（Forward Secrecy）：即使攻擊者未來破解了 PSK，也無法解密過去截獲的連線資料，因為每次連線的 PMK 都是獨立產生的。
（2）線上限速防護：SAE 要求每次認證嘗試都必須與 AP 進行即時互動，無法進行離線暴力破解，攻擊者每次嘗試都需要實際連接 AP，速率受到大幅限制。
OWE（Opportunistic Wireless Encryption）
針對開放式無線網路（咖啡廳、機場 Wi-Fi），OWE 讓裝置在無需密碼的情況下，透過 Diffie-Hellman 金鑰交換建立加密通道。用戶無需輸入密碼，但資料已被加密，防止被動竊聽。OWE 不提供認證（任何人仍可連接），但解決了開放網路的最大安全問題——明文傳輸。
強化企業模式（WPA3-Enterprise 192-bit）
為高安全需求場景（政府、金融）提供 192 位元安全套件：使用 GCMP-256 加密（取代 CCMP-128）、SHA-384 完整性驗證、ECDSA-384 伺服器憑證，提供比 WPA2-Enterprise 更高的安全強度。
管理幀保護強制（802.11w PMF）
WPA3 強制要求啟用 PMF（Protected Management Frames，802.11w），保護 Deauthentication 和 Disassociation 管理訊框不被偽造。WPA2 環境中 PMF 是可選的，WPA3 將其設為必要條件。

WPA3 的已知弱點

WPA3 發布後不久，研究人員（Mathy Vanhoef）即發現 SAE 握手中存在Dragonblood 系列漏洞（計時旁道攻擊和快取旁道攻擊），可降低攻擊成本。Wi-Fi Alliance 已於 2019 年修訂 SAE 規範並要求廠商更新，現代 WPA3 實作已修補這些問題。

WPA2 / WPA3 混合模式（Transition Mode）

在 WPA3 普及初期，許多環境需要同時支援 WPA2 和 WPA3 裝置。AP 可設定為WPA2/WPA3 混合模式（Transition Mode），支援 WPA3 的裝置自動使用 SAE，舊裝置繼續使用 WPA2-PSK。混合模式下的安全強度介於兩者之間，建議隨著舊裝置汰換，逐步移轉至純 WPA3 模式。

6.7Personal vs Enterprise 模式

WPA2 和 WPA3 各自有兩種認證模式——Personal（個人模式）和Enterprise（企業模式），兩者的差異不在於加密強度，而在於身分認證方式。

比較項目Personal（PSK）Enterprise（802.1X）
認證方式 共享預設金鑰（密碼） 個人憑證、帳號密碼或憑證（透過 RADIUS）
個別用戶識別 無（所有人共用同一密碼） 有（每人有獨立帳號，可稽核到個人）
離職撤銷 需更換全部裝置密碼 停用該用戶帳號即可，秒速撤銷
密碼洩漏風險 高（共用密碼一旦洩漏全員受影響） 低（個人憑證洩漏不影響他人）
動態 VLAN 分配 不支援 支援（RADIUS 屬性動態指定 VLAN）
部署複雜度 低（AP 設定一個密碼即可） 高（需要 RADIUS 伺服器、PKI 憑證架構）
適用場景 家庭、小型辦公室、訪客網路 中大型企業、學術機構、醫療機構
安全等級 中（WPA3 下可接受） 高（每用戶獨立金鑰，最佳實務）
企業環境的最佳實踐

任何需要個人身分認證、存取稽核，或人員流動率較高的環境，都應使用 WPA2/WPA3-Enterprise 搭配 802.1X 認證。Personal 模式的共用密碼無法追溯到個人行為，且離職員工的存取無法精準撤銷（不更換密碼的話）。

IEEE 802.1X 是一個基於埠的網路存取控制（Port-Based Network Access Control）標準，最初為有線乙太網路設計，後被 WPA2/WPA3-Enterprise 採用作為無線認證框架。它定義了三個參與角色和它們之間的通訊協定。

三個角色的職責

Supplicant（申請者）
要求存取網路的客戶端裝置。裝置上必須執行 EAP 客戶端軟體，Windows / macOS / iOS / Android 均內建 802.1X 用戶端。Supplicant 負責提交認證憑據（帳號密碼或數位憑證）。
Authenticator（認證者）
控制網路存取的設備，無線環境中通常是 AP 或 WLC。Authenticator 本身不驗證身分，只是在 Supplicant（使用 EAPoL）和 Authentication Server（使用 RADIUS）之間充當轉發代理。在認證完成前，AP 只允許 EAPoL 認證流量通過；認證成功後才開放資料流量。
Authentication Server（認證伺服器）
實際執行身分驗證邏輯的伺服器，通常是 RADIUS（Remote Authentication Dial-In User Service）伺服器。RADIUS 伺服器根據帳號資料庫（本機資料庫、Active Directory、LDAP）驗證 Supplicant 的憑據，並回傳 Access-Accept（允許）或 Access-Reject（拒絕）。

6.9EAP 類型詳解

EAP（Extensible Authentication Protocol，可延伸認證協定）是 802.1X 使用的認證協定框架，支援多種不同的認證方法（稱為 EAP 類型），各有不同的安全強度和部署複雜度。

EAP 類型認證方式伺服器憑證客戶端憑證安全強度常見用途
EAP-MD5 帳號 + 密碼（MD5 雜湊） 否 否 極低（無法防 MITM） 僅有線環境，無線不應使用
LEAP（Cisco） 帳號 + 密碼 否 否 低（字典攻擊風險） 已廢棄，勿使用
EAP-PEAP 外層 TLS 通道 + 內層帳號密碼（MSCHAPv2） 需要 否 中高 企業 Windows 環境最常見，整合 AD
EAP-TTLS 外層 TLS 通道 + 內層多種方式（PAP/CHAP/MSCHAPv2） 需要 否 中高 跨平台環境，macOS / Linux 常用
EAP-TLS 雙向 TLS 憑證（伺服器 + 客戶端各需憑證） 需要 需要 最高 高安全企業、搭配 PKI 憑證架構
EAP-FAST（Cisco） PAC（Protected Access Credential）取代憑證 可選 否 中高 Cisco 設備環境，簡化憑證部署

最常用的兩種 EAP：PEAP vs EAP-TLS

EAP-PEAP
部署最廣泛的企業 EAP 方式，因為只需要 RADIUS 伺服器安裝 TLS 憑證，客戶端裝置不需要個人憑證，只需輸入帳號密碼（通常整合至 Active Directory）。Windows 裝置加入網域後可自動取得 Wi-Fi 連線設定，使用者無感知。

安全注意事項：客戶端裝置必須設定驗證 RADIUS 伺服器的 TLS 憑證（CA 憑證），否則攻擊者可架設偽造 RADIUS 伺服器進行中間人攻擊（Evil Twin + Rogue RADIUS），截取帳號密碼。
EAP-TLS
安全強度最高的 EAP 方式，使用雙向憑證認證（伺服器和客戶端各有獨立的 X.509 憑證）。即使攻擊者取得用戶的密碼也無法連線，因為連線需要私鑰（儲存在裝置的 TPM 晶片或智慧卡中）。

部署挑戰：需要完整的 PKI（公開金鑰基礎設施）架構，為每台裝置簽發客戶端憑證，並有憑證到期和撤銷（CRL / OCSP）的管理需求，部署複雜度較高。

6.10RADIUS 伺服器部署

RADIUS（Remote Authentication Dial-In User Service）伺服器是 802.1X 認證架構的核心，負責實際驗證用戶身分並回應 AP/WLC 的認證請求。

常見 RADIUS 伺服器選擇

RADIUS 軟體類型後端整合授權適用場景
Microsoft NPS Windows Server 內建 Active Directory（原生整合） Windows Server 授權 已有 AD 環境的 Windows 企業
FreeRADIUS 開源 LDAP、AD、資料庫、PAM 免費（GPL） 技術能力強的組織，最靈活
Cisco ISE 商業（全功能 NAC） AD、LDAP、MDM 商業授權 大型企業，需要裝置合規（Posture）
Aruba ClearPass 商業（全功能 NAC） AD、LDAP、MDM、社群帳號 商業授權 多廠商混合環境，進階政策管理
雲端 RADIUS（JumpCloud、Foxpass） SaaS LDAP、AD、Google Workspace 訂閱制 無 IT 人力維護伺服器的中小企業

RADIUS 基本設定（FreeRADIUS + EAP-PEAP 範例）

# /etc/freeradius/3.0/clients.conf — 定義信任的 AP / WLC
client AP-WLC {
    ipaddr  = 192.168.99.10    # AP 或 WLC 的 IP
    secret  = SharedSecret123! # AP 與 RADIUS 間的共享金鑰（需夠長）
    nastype = cisco
}

# /etc/freeradius/3.0/users — 本機帳號（生產環境建議改用 AD/LDAP）
alice   Cleartext-Password := "P@ssw0rd123!"
        Reply-Message = "Welcome, Alice",
        Tunnel-Type = VLAN,
        Tunnel-Medium-Type = IEEE-802,
        Tunnel-Private-Group-Id = 10   # 指定 VLAN 10

# /etc/freeradius/3.0/mods-enabled/eap — EAP 設定
eap {
    default_eap_type = peap
    peap {
        default_eap_type = mschapv2
        tls-config tls-common  # 使用 TLS 設定區段
    }
    tls-config tls-common {
        certificate_file = /etc/ssl/certs/radius.crt
        private_key_file = /etc/ssl/private/radius.key
        ca_file          = /etc/ssl/certs/ca.crt
    }
}

RADIUS 高可用性設計

RADIUS 伺服器是無線認證的單點故障：若 RADIUS 無法回應，所有需要 802.1X 認證的裝置都無法連線。生產環境必須部署：

  • 主備雙 RADIUS 伺服器：AP/WLC 設定 Primary 和 Secondary RADIUS，主伺服器無回應時自動切換
  • 不同實體位置：兩台 RADIUS 伺服器放在不同機房或不同子網，避免單一網路故障同時影響兩台
  • 定期測試容錯：定期測試切換到 Secondary RADIUS 的行為是否正常，不要等到緊急時才發現問題

6.11動態 VLAN 分配

動態 VLAN 分配（Dynamic VLAN Assignment）是 802.1X 最強大的功能之一：RADIUS 伺服器在認證成功後，透過 RADIUS 屬性告知 AP，讓此用戶連接到特定 VLAN，無需為每個群組建立獨立的 SSID。

動態 VLAN 的工作原理

動態 VLAN 分配流程： 用戶 Alice（員工）連接 "CorpWiFi" SSID → 802.1X 認證 → RADIUS 查詢 Alice 的帳號屬性 → RADIUS 回傳 Access-Accept： Tunnel-Type = VLAN Tunnel-Medium-Type = 802 Tunnel-Private-Group-Id = 10 ← 員工 VLAN → AP 將 Alice 的流量放入 VLAN 10 用戶 Bob（承包商）連接同一個 "CorpWiFi" SSID → 802.1X 認證 → RADIUS 查詢 Bob 的帳號屬性 → RADIUS 回傳 Access-Accept： Tunnel-Private-Group-Id = 50 ← 承包商 VLAN（受限存取） → AP 將 Bob 的流量放入 VLAN 50 結果：相同 SSID，不同用戶 → 不同 VLAN → 不同存取權限 無需為員工和承包商建立兩個不同的 SSID
Fig 6.4 — 動態 VLAN 分配：單一 SSID 多 VLAN 隔離

RADIUS 動態 VLAN 屬性

# RADIUS Access-Accept 回應中包含的三個必要屬性

Tunnel-Type           = VLAN      # RFC 2868 屬性，值 = 13
Tunnel-Medium-Type    = 802       # RFC 2868 屬性，值 = 6（IEEE 802）
Tunnel-Private-Group-Id = "10" # VLAN ID，字串或整數皆可

# 在 FreeRADIUS users 檔案中，用戶屬性設定範例：

alice   Cleartext-Password := "P@ss!"
        Tunnel-Type = VLAN,
        Tunnel-Medium-Type = IEEE-802,
        Tunnel-Private-Group-Id = 10

contractor_bob  Cleartext-Password := "B0bPass"
                Tunnel-Type = VLAN,
                Tunnel-Medium-Type = IEEE-802,
                Tunnel-Private-Group-Id = 50
與 AD 群組整合

在企業環境中，動態 VLAN 通常與 Active Directory 群組整合：RADIUS 伺服器（如 Microsoft NPS 或 Cisco ISE）根據用戶所在的 AD 群組自動決定分配的 VLAN。例如「Sales 群組 → VLAN 20」、「IT 群組 → VLAN 10」，管理員只需維護 AD 群組成員，VLAN 分配自動更新。

6.12常見無線攻擊手法

了解攻擊手法是制定防禦策略的前提。以下是無線網路環境中最常見的攻擊類型：

偽裝攻擊
Evil Twin（惡意雙胞胎）
攻擊者架設與合法 AP 相同 SSID 的惡意 AP，並以更強的訊號吸引裝置連接。裝置連接後，攻擊者成為中間人，可竊聽所有未加密流量或呈現釣魚頁面。
使用 802.1X 認證（裝置必須驗證 RADIUS 伺服器憑證），惡意 AP 無法提供有效憑證；部署 WIDS 偵測重複 SSID；802.11w 防止管理訊框偽造。
阻斷服務
Deauth Flood（反認證洪水）
802.11 的 Deauthentication 管理訊框在 WPA2 中未加密且無需認證，攻擊者可持續偽造傳送，強迫所有客戶端斷線，造成持續的無線服務中斷。
強制啟用 802.11w（PMF，Protected Management Frames）——WPA3 強制要求，WPA2 需手動啟用。PMF 加密管理訊框，使偽造的 Deauth 封包被裝置忽略。
密碼攻擊
WPA2 握手擷取 + 字典攻擊
攻擊者使用 aircrack-ng 等工具擷取 WPA2 四向握手封包（可透過傳送 Deauth 封包強制裝置重新連線），再用 hashcat + GPU 對 PMK 進行離線暴力 / 字典攻擊。
使用 WPA3-SAE（根本解決離線攻擊問題）；WPA2 環境中使用隨機長密碼（20+ 字元）；定期輪換 PSK；企業環境改用 802.1X。
竊聽攻擊
被動竊聽（Passive Eavesdropping）
在開放 Wi-Fi（無密碼）或使用弱加密的網路中，攻擊者被動收集空中所有封包，還原 HTTP、DNS、SMTP 等未加密協定的明文內容。
開放網路使用 WPA3-OWE 提供機會性加密；應用層使用 HTTPS / TLS；推廣 VPN 使用（特別是在公共 Wi-Fi）；鼓勵 DNS-over-HTTPS。
未授權接入
Rogue AP（惡意存取點）
內部人員（或入侵者）在企業內網插入未授權的 AP（或將筆電設為 Wi-Fi 熱點），繞過防火牆和 NAC 政策，建立未受管控的網路入口。
部署 WIDS 持續掃描未授權 AP；有線網路啟用 802.1X 埠認證防止未授權裝置插入；定期無線稽核；在敏感區域停用交換器埠。
重放攻擊
PMKID / KRACK 攻擊
PMKID 攻擊無需完整握手，只需從 AP 的 Beacon 或 Probe Response 中提取 PMKID 值，再對其進行離線字典攻擊。KRACK 則重放四向握手訊息 3 使金鑰歸零。
確認 AP 和客戶端韌體已更新（KRACK 補丁）；升級至 WPA3（SAE 根本防止 PMKID 攻擊）；使用複雜 PSK 減少字典攻擊成功率。

6.13無線入侵偵測（WIDS）

WIDS（Wireless Intrusion Detection System，無線入侵偵測系統）持續掃描無線環境，偵測並告警各種安全威脅和政策違規事件，是企業無線安全架構的重要組成。

WIDS 可偵測的威脅類型

威脅類型偵測方式回應動作
Rogue AP（未授權 AP） 掃描到 BSSID 不在白名單內的 AP，或廣播與合法 SSID 相同的 AP 告警 + 定位 + 自動有線側封鎖（Wired Containment）
Evil Twin 偵測到與合法 AP 相同 SSID 但不同 BSSID 的 AP 廣播 立即告警，記錄位置，通知管理員
Deauth / Disassoc Flood 短時間內大量 Deauthentication 訊框，超過門檻值 告警 + 記錄來源 MAC，啟用 PMF 防護
IBSS（Ad-hoc）裝置 偵測到裝置發出 IBSS Beacon（非 AP 基礎架構模式） 告警，可自動傳送 Deauth 阻止連線
WPA2 握手擷取嘗試 偵測大量針對特定裝置的 Deauth 封包（主動觸發握手） 告警，記錄攻擊者 MAC
弱加密使用 掃描到使用 WEP 或 TKIP 的 AP 或客戶端 告警，要求升級安全設定

WIDS 的兩種工作模式

Monitor AP（專用監控 AP）
部分 AP 設定為純監控模式，不提供用戶服務，全時段掃描所有頻道。覆蓋最全面，但需要額外 AP 成本。適合高安全需求環境（醫院、金融機構）。
Hybrid WIDS（服務 + 監控）
普通服務 AP 在服務用戶的同時，利用空閒時間輪流切換至其他頻道掃描。不需要額外硬體，但掃描覆蓋率和即時性不如專用監控 AP。現代 AP 通常可設定掃描週期。

6.14管理幀保護（802.11w / PMF）

標準 802.11 的管理訊框（Deauthentication、Disassociation、Action 等）長期以來都是未加密、未認證的，任何人都可以偽造發送。IEEE 802.11w（PMF，Protected Management Frames，管理幀保護）於 2009 年正式成為標準，為這些關鍵管理訊框提供加密和完整性保護。

PMF 的保護範圍

受 PMF 保護的訊框
Deauthentication、Disassociation、Action（包含 QoS、Spectrum Management、SA Query）。這些訊框加上 MIC 後，接收端可驗證訊框真實性，偽造的 Deauth 封包會被忽略。
不受 PMF 保護的訊框
Beacon、Probe Request / Response、Authentication（認證前的訊框）。這些訊框在認證建立前就需要傳輸，PMF 的金鑰尚未協商完成，故無法保護。

PMF 的三種模式

模式AP 行為客戶端支援要求適用場景
Disabled（停用） 不提供 PMF，管理訊框未受保護 無要求 不建議，相容舊裝置用
Optional（可選） AP 支援 PMF，但不強制客戶端使用 支援 PMF 的裝置自動使用，不支援的也可連線 過渡期，逐步啟用
Required（必要） AP 強制要求所有客戶端必須支援 PMF 才能連線 必須支援 PMF WPA3 環境、高安全需求（建議）
WPA3 強制要求 PMF

WPA3 認證設備必須以 Required 模式啟用 PMF，從根本防止 Deauth Flood 攻擊。若在 WPA2/WPA3 混合模式下，建議至少設定 PMF 為 Optional，讓支援 PMF 的 WPA3 裝置獲得保護，同時維持 WPA2 裝置的相容性。

Chapter 06 Summary
  • 無線安全三要素：認證（誰可以連）、加密（資料不被竊聽）、完整性（資料不被篡改）。
  • WEP 使用 RC4 + 24-bit 靜態 IV，數小時內可破解，2004 年廢棄，嚴禁使用。WPA-TKIP 同樣已廢棄，避免使用。
  • WPA2 使用 AES-128 + CCMP，安全性大幅提升；但 WPA2-Personal（PSK）的四向握手可被擷取後進行離線字典攻擊，密碼必須夠強（20 字元以上隨機）。
  • WPA3 引入 SAE（Dragonfly）握手，提供前向保密並防止離線字典攻擊；OWE 為開放網路提供機會性加密；PMF 成為強制要求。
  • Personal 模式使用共用 PSK，無法追溯個人；Enterprise 模式透過 802.1X + RADIUS 實現個人認證、動態 VLAN 和即時撤銷。
  • 802.1X 三角色：Supplicant（裝置）、Authenticator（AP/WLC，轉發）、Authentication Server（RADIUS，驗證）。
  • EAP-TLS（雙向憑證）安全性最高；EAP-PEAP（帳號密碼 + 伺服器憑證）部署最廣泛；PEAP 環境必須驗證 RADIUS 伺服器憑證防 MITM。
  • 動態 VLAN 讓 RADIUS 根據用戶身分決定分配的 VLAN，同一 SSID 可服務不同權限等級的用戶群體。
  • 常見攻擊：Evil Twin（偽造 AP）、Deauth Flood（DoS）、WPA2 握手字典攻擊、PMKID 攻擊、Rogue AP（內部插入）。
  • 802.11w PMF 加密管理訊框，防止 Deauth Flood；WIDS 持續掃描偵測惡意 AP 和攻擊行為；WPA3 強制啟用 PMF。
