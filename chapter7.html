7.1無線 QoS 的必要性

QoS（Quality of Service，服務品質）是網路依各應用的時間敏感程度給予差異化服務保障的能力。沒有 QoS 的網路採用盡力而為（Best-Effort）模式——所有封包一視同仁地排隊，先到先服務。這在網路空閒時問題不大，但多用戶競爭有限無線頻道時便產生嚴重影響。

【無 QoS】用戶 A: VoIP | 用戶 B: 大型下載 | 用戶 C: 4K 串流 所有封包平等競爭 → 大流量佔多數傳輸機會 → VoIP 封包被延遲 → 通話斷斷續續、機器音、中斷 【有 WMM QoS】 VoIP 封包進入最高優先 AC_VO 佇列 → 語音優先取得傳輸機會（短 AIFSN + 小 CW） → 大流量退讓 → 通話清晰穩定
Fig 7.1 — QoS 在多流量競爭場景的效果對比

無線介質是共享廣播介質，所有同頻道裝置共用同一無線通道，QoS 需在 MAC 層的競爭機制（CSMA/CA）中嵌入優先級邏輯，比有線 QoS 更為複雜。

7.2802.11e 與 WMM

IEEE 802.11e（2005 年）定義了EDCA（Enhanced Distributed Channel Access）機制，將單一 CSMA/CA 競爭佇列擴展為四個優先級佇列（Access Category，AC）。Wi-Fi Multimedia（WMM）是 Wi-Fi Alliance 基於 802.11e 的認證計畫，確保不同廠商裝置間的 QoS 互通性，現代 Wi-Fi 設備幾乎全面支援且預設啟用。

AC_VO
語音
最高優先，最短等待時間。VoIP 通話、緊急廣播。DSCP EF（46）／ 802.1p 6–7
AC_VI
視訊
次高優先，低延遲低抖動。視訊會議、串流媒體。DSCP AF41（34）／ 802.1p 4–5
AC_BE
盡力而為
預設優先，一般資料流量。網頁、電郵、檔案傳輸。DSCP BE（0）／ 802.1p 0、3
AC_BK
背景
最低優先，允許較高延遲。備份作業、P2P 下載。DSCP CS1（8）／ 802.1p 1–2
WMM 的限制

WMM 只保證相對優先，不是絕對頻寬保障。WMM 只在同一台 AP範圍內有效；端到端 QoS 還需有線交換器配合 DSCP / 802.1p 標記。

7.3EDCA 競爭參數

EDCA 透過三個核心參數控制每個 AC 佇列的競爭優先級，數值越小代表優先級越高：

AIFSN
Arbitration Interframe Space Number：頻道空閒後需等待的最小時隙數。越小越快取得傳輸機會。AC_VO = 2（最短）；AC_BK = 7（最長）。
CWmin / CWmax
Contention Window（競爭視窗）：等待 AIFSN 後，在此範圍內隨機選退避時間（Backoff）。越小期望等待越短。失敗後 CW 加倍直到 CWmax。AC_VO：CWmin = 3；AC_BK：CWmin = 15。
TXOP
Transmission Opportunity：取得傳輸機會後允許連續傳輸的最大時間。AC_VO = 1.504 ms；AC_BE = 0（每次只能傳一個訊框）。

7.5頻寬限制與流量整形

AP 提供頻寬限制（Rate Limiting）功能，防止單一裝置或訪客網路佔用過多資源。

Per-SSID 限制
對整個 SSID 流量設上限。例如訪客 SSID 總頻寬 50 Mbps，有效控制訪客對企業網際網路出口的佔用。
Per-Client 限制
對每台裝置個別設定上下行上限，如訪客下行 20 Mbps / 上行 5 Mbps，防止單一用戶大量下載影響其他人。
應用層流量整形
進階平台（Meraki、Aruba）可識別應用類型（YouTube、P2P BitTorrent），針對特定應用限速或封鎖，Teams 視訊不受限而 P2P 限速至 1 Mbps。

7.6無線漫遊基礎

在 ESS 環境中，裝置從一台 AP 移動到另一台 AP 的過程稱為漫遊（Roaming）。802.11 標準將漫遊決策完全交給客戶端裝置，AP 無法強制切換（除非使用 802.11v）。客戶端通常在 RSSI 低於 -70 到 -75 dBm、封包遺失率超過門檻，或收到 802.11v BSS Transition Request 時觸發漫遊。

Probe / Auth / Assoc │ ~20 ms │ │ 3. WPA2 認證（PSK 或 802.1X） │ 50–300 ms ★瓶頸 │ │ 4. 四向握手（PTK 協商） │ ~20 ms │ ├─────────────────────────────────────────────────────┤ │ 總計（PSK） │ ~150–550 ms │ │ 總計（802.1X） │ ~300–600 ms │ └─────────────────────────────────────────────────────┘ ★ 802.1X 需與 RADIUS 完成完整 EAP 交換，是最大延遲來源 VoIP 可接受的漫遊延遲上限：< 50 ms

7.7802.11r 快速 BSS 切換

IEEE 802.11r（FT, Fast BSS Transition）透過預先在目標 AP 快取金鑰材料，讓裝置漫遊時跳過耗時的完整認證流程，直接進行四向握手，將漫遊時間壓縮至 50 ms 以下。

# 802.11r FT 金鑰階層
PMK-R0（頂層金鑰）
  └── 由 PSK 或 802.1X 認證產生，儲存於 WLC（R0KH）

PMK-R1（中間金鑰）
  └── 由 PMK-R0 + 目標 AP 的 R1KH-ID 衍生
  └── 漫遊時由 WLC 下推至目標 AP

PTK（成對暫時金鑰）
  └── 由 PMK-R1 + 四向握手推導（每次連線獨立）

漫遊時：目標 AP 已有 PMK-R1 → 直接四向握手（~10 ms）
→ 省去重新 RADIUS 認證（最耗時步驟）
模式一
Over-the-Air FT
流程直接與目標 AP 交換 FT Auth
漫遊時間< 20 ms
特點訊框數最少，延遲最低
模式二
Over-the-DS FT
流程FT Auth 透過有線 DS（WLC）轉發
漫遊時間< 50 ms
特點裝置無需先掃描到目標 AP
802.11r 相容性問題

極少數舊型 Android 和部分 IoT 裝置在啟用 802.11r 後會無法連線。現代 AP 通常提供 Adaptive FT（自適應）模式：支援 FT 的裝置使用快速漫遊，不支援的退回標準漫遊，兩者共存。

7.8802.11k 無線資源量測

IEEE 802.11k（Radio Resource Management）讓 AP 主動向客戶端提供鄰居 AP 清單（Neighbor Report），幫助客戶端在漫遊時快速找到候選 AP，省略耗時的全頻道掃描。

【沒有 802.11k】 裝置決定漫遊 → 掃描全部頻道（2.4G 13個 + 5G 20+個） → 找到鄰居 AP → 漫遊 掃描時間：50～200 ms（期間停止資料傳輸） 【有 802.11k】 裝置連線期間向 AP1 請求 Neighbor Report： AP1 回應：[AP2: CH36, RSSI -60] [AP3: CH40, RSSI -68] 裝置需要漫遊時 → 只掃描報告中指定的頻道 → 立即找到目標 AP → 漫遊 掃描時間：~10 ms（縮短 80% 以上）
Fig 7.3 — 802.11k 縮短漫遊掃描時間的效益
Beacon Report
客戶端回報附近其他 AP 的 BSSID、RSSI 和頻道資訊，讓 WLC 掌握即時無線拓撲，輔助 RRM 決策。
Neighbor Report
AP 回應客戶端的鄰居清單請求，提供可漫遊目標 AP 的詳細資訊（BSSID、頻道、AP 能力）。
Channel Load Report
量測特定頻道的使用率（Busy Time / Total Time），輔助頻道規劃和干擾分析。

7.9802.11v BSS 切換管理

IEEE 802.11v（BSS Transition Management）賦予 AP 主動向客戶端建議或要求切換到另一台 AP 的能力，是解決Sticky Client（黏性客戶端）問題的主要工具。

建議模式
AP 送出 BSS Transition Request，附上候選 AP 清單。客戶端可接受或拒絕，決策權仍在客戶端手中。
強制模式（Disassociation Imminent）
AP 設定 Disassociation Imminent 位元，告知客戶端 AP 將在指定時間後斷開連線，客戶端必須主動切換，否則被強制斷線。這是解決 Sticky Client 最有效的手段。
應用場景AP 的動作目的
Sticky Client裝置 RSSI < -75 dBm，送出 Disassociation Imminent + 候選清單強制訊號差的裝置切換至更近的 AP
AP 負載均衡AP 連線數超過門檻，向部分裝置送出遷移建議引導至負載較低的相鄰 AP
Band Steering 輔助雙頻裝置連在 2.4 GHz，送出 BSS Transition 指向 5 GHz BSSID引導裝置至更高效能的 5 GHz
AP 計劃維護維護前送出 BSS Transition Request，讓裝置提前切換最小化維護時的用戶中斷時間

7.10漫遊三劍客整合效果

802.11k、802.11r、802.11v 各自解決漫遊問題的不同面向，三者同時啟用才能發揮最大效果：

標準解決的核心問題主要機制效果
802.11k漫遊前「找哪裡」鄰居報告（Neighbor Report）掃描時間縮短 60～80%
802.11r漫遊中「認證耗時」預快取金鑰（FT Key Hierarchy）漫遊時間 <50 ms（802.1X 從 500 ms 降至 30 ms）
802.11v漫遊時機「黏著不動」BSS Transition Request主動引導 Sticky Client 及時切換
三劍客協同作業的完整漫遊時序： t=0 ms：裝置 RSSI 下降至 -72 dBm（802.11k 已取得鄰居清單） t=5 ms：AP 啟動 802.11v → 送出 BSS Transition Request + 候選清單（AP2: CH36） t=8 ms：裝置決定漫遊至 AP2 t=10 ms：只掃描 CH36（802.11k 指定）→ 立即找到 AP2 t=12 ms：802.11r FT Authentication → AP2 從 WLC 取得 PMK-R1 t=25 ms：四向握手完成，PTK 建立 t=28 ms：裝置關聯至 AP2，資料傳輸恢復 總漫遊時間：~28 ms（VoIP 通話不受影響） 無三劍客標準漫遊：150～600 ms → 通話斷音、機器音、中斷

7.11效能瓶頸識別

無線網路效能問題可能來自多個層面，系統性地識別瓶頸所在是優化的第一步：

1
有線端還是無線端？
用有線裝置直連交換器做速率測試（iPerf3）。有線正常 → 問題在無線端；有線也慢 → 問題在上行網路（ISP 或骨幹）。確認此步驟可排除約 30% 的「Wi-Fi 問題」實際上是有線端問題。
2
確認 RSSI 和 MCS 索引
在 AP 管理介面查看問題裝置的 RSSI（應 >-70 dBm）和 MCS 索引（越高越好）。RSSI <-75 dBm → 覆蓋不足；RSSI 正常但 MCS 低 → 干擾或低 SNR 問題。
3
確認頻道使用率和重傳率
頻道使用率超過 75% → 頻道過載，調整頻道規劃或增加 AP；重傳率超過 20% → 訊號品質差或干擾，需頻譜分析找出干擾源。
4
確認 AP 並發裝置數與 CPU
連線裝置超過 50 台或 CPU 持續 >80% → AP 過載，需增加 AP、調整 Band Steering，或更換高階 AP。
5
是否有低速裝置拖累整體速率？
802.11b/g 舊裝置接入時，AP 必須降低整體速率以相容（Protection Mechanism）。考慮停用低速率（見 7.14 節）或隔離至專用 SSID。

7.12空氣時間公平性

傳統的封包公平（Packet Fairness）在混合速率環境中會造成嚴重效能不公平：Wi-Fi 6 裝置傳一個封包只需 0.01 ms，而 Wi-Fi 4 裝置需要 0.15 ms——舊裝置佔用了 15 倍空氣時間卻只傳了同樣數量的封包，嚴重拖累整體效率。

【封包公平（傳統）】 Wi-Fi 6 裝置 A（600 Mbps）：1 封包 → 0.01 ms 空氣時間 Wi-Fi 4 裝置 B（54 Mbps） ：1 封包 → 0.15 ms 空氣時間 結果：裝置 B 佔用 15 倍空氣時間，舊裝置嚴重拖累整體效率。 【空氣時間公平（Airtime Fairness）】 Wi-Fi 6 裝置 A：10 ms → 傳送約 750 KB 資料 Wi-Fi 4 裝置 B：10 ms → 傳送約 67 KB 資料 結果：快速裝置充分發揮效能，舊裝置不再拖累整體網路。
Fig 7.5 — 封包公平 vs 空氣時間公平的差異

Airtime Fairness 在混合速率環境（辦公室新舊裝置共存）中效益顯著。UniFi 稱為「Airtime Fairness」；Cisco WLC 稱為「802.11e Airtime Fairness」。在無線電設定中啟用即可。

7.13Rate Adaptation 速率自適應

Rate Adaptation（速率自適應）是 AP 和客戶端根據當前無線鏈路品質，動態調整 MCS 索引的機制，在訊號品質和速率之間取得最佳平衡。

# 速率自適應決策邏輯（簡化）
SNR > 30 dB  → 高 MCS（MCS 11, 1024-QAM）→ 最高速率
SNR 20–30 dB → 中 MCS（MCS 7, 64-QAM）   → 中等速率
SNR 10–20 dB → 低 MCS（MCS 3, QPSK）     → 保守速率
SNR < 10 dB  → 最低 MCS（MCS 0, BPSK）   → 最低速率

動態調整：
  ↑ 連續成功傳輸後嘗試提升 MCS
  ↓ 重傳率超過門檻後立即降低 MCS
Minstrel
Linux mac80211 驅動常用。隨機探測不同 MCS 的成功率，統計找出最佳速率。能快速適應環境，但在高干擾環境有額外開銷。
CARA
Collision-Aware Rate Adaptation：區分傳輸失敗是碰撞（Collision）還是訊號差，避免因碰撞誤判而過度降速，在高密度環境表現更穩定。
廠商專有演算法
Cisco、Qualcomm 等廠商實作各自優化的速率自適應演算法，通常整合功率控制和空間串流選擇，整體效果優於通用開源實作。

啟用 WMM QoS確認 WMM 啟用，配置正確的 DSCP 信任策略VoIP / 視訊流量優先，通話品質提升中（VoIP 環境為高）
優化 AP 發射功率依部署密度調整功率，避免過高造成同頻干擾減少相鄰 AP 干擾，整體效能提升中
設定 Minimum RSSIAP 拒絕 RSSI 低於 -80 dBm 的裝置連線排除訊號極差的遠端裝置，保護整體頻道品質中
啟用 BSS Coloring確認 Wi-Fi 6 AP 的 BSS Color 功能啟用（通常預設啟用）空間重用效率提升，減少不必要的頻道退讓低至中（Wi-Fi 6 環境）

7.15故障排除方法論

無線故障排除的核心是系統性地縮小問題範圍，而非憑直覺猜測。OSI 模型由下而上是最可靠的框架：

Layer 1（實體層）
  ✓ AP 是否有 PoE 供電（LED 狀態確認）
  ✓ 網路線是否連接正常（交換器埠 Link 燈）
  ✓ 無線訊號是否可被偵測（RSSI 合理值）
  ✓ 頻道是否有干擾（頻譜分析儀）

Layer 2（資料鏈結層）
  ✓ AP 是否成功連接 WLC / 雲端控制器
  ✓ 客戶端是否成功關聯到 AP
  ✓ WPA2/3 認證是否成功（無認證失敗日誌）
  ✓ VLAN 標記是否正確（交換器 Trunk 設定）

Layer 3（網路層）
  ✓ AP 和客戶端是否取得 IP（DHCP 成功）
  ✓ 客戶端能否 Ping 閘道器
  ✓ 防火牆規則是否阻擋特定流量

Layer 4–7（傳輸到應用層）
  ✓ DNS 解析是否正常（nslookup 測試）
  ✓ 特定應用埠是否被封鎖
  ✓ QoS 標記是否正確（Wireshark 抓包確認）
問題範圍
單一用戶 → 裝置端；某 AP 附近用戶 → AP 問題；全部用戶 → WLC 或上行網路問題。
問題時間
問題何時開始？是否有系統變更剛好在問題出現前？間歇性問題多與負載或干擾有關。
具體現象
「連線不上」（認證）、「速率慢」（效能）、「頻繁斷線」（漫遊或穩定性）——不同現象指向不同排查方向。
頻段測試
強制裝置分別連接 2.4 GHz 和 5 GHz 測試。只有 2.4 GHz 差 → 2.4 GHz 干擾；兩個頻段都差 → AP 上行或認證問題。

7.16VoIP 無線品質排障

VoIP 是對無線品質最敏感的應用，常作為無線網路品質的「金絲雀指標」——VoIP 通話品質好，代表無線整體狀況良好。

MOS 分數通話品質感知對應網路條件
4.0 – 5.0優秀，接近面對面延遲 <50 ms，抖動 <10 ms，封包遺失 <0.1%
3.5 – 4.0良好，可接受延遲 <100 ms，抖動 <20 ms，封包遺失 <1%
3.0 – 3.5普通，稍有不清晰延遲 <150 ms，抖動 <30 ms，封包遺失 <3%
2.0 – 3.0差，明顯干擾通話高延遲或抖動，頻繁封包遺失
< 2.0無法接受嚴重網路問題
Q
通話有斷音 / 靜音片段
最常見原因：漫遊切換。確認 802.11r/k/v 是否啟用，用 AP 日誌確認漫遊時間是否超過 50 ms。靜態用戶若有斷音，則排查封包遺失率和 WMM AC_VO 佇列是否正常運作。
Q
通話有機器音 / 機械感
最常見原因：抖動（Jitter）過高。VoIP 使用抖動緩衝器平滑延遲變化，抖動超過緩衝器容量時封包順序錯亂導致機器音。確認 WMM AC_VO 是否正確啟用，RTP 封包的 DSCP 是否標記為 EF（46）。
Q
通話有回音 / 迴響
通常不是無線問題。回音多數由電話設備的回音消除（AEC）失效、揚聲器設定或會議室聲學環境造成。確認電話機或軟體客戶端的 AEC 設定，排查環境噪音反射問題。
Q
特定時段或位置品質差
時段相關：確認尖峰時段頻道使用率（應 <75%）和 WMM 是否優先語音流量；必要時增加 AP 或調整頻道規劃。位置相關：現場量測 RSSI（應 >-70 dBm），確認是否在 AP 覆蓋邊緣或有干擾源。

7.17診斷工具與指令

iPerf3
效能量測
量測 TCP / UDP 吞吐量、延遲和抖動。客戶端和伺服器各執行，測量無線到有線的實際速率，是最基本的效能確認工具。
Wireshark
封包分析
擷取和分析 802.11 訊框，驗證 WMM DSCP 標記、四向握手、漫遊事件、重傳率。需支援 Monitor Mode 的 Wi-Fi 網卡。
Ekahau AI Pro
無線勘查
專業 Site Survey 工具，量測 RSSI 熱圖、頻道干擾、SNR 分布，生成視覺化報告。業界最廣泛使用的無線診斷工具之一。
NetSpot（免費）
無線勘查
免費版可做基本被動式 Survey，掃描鄰居 AP 並生成覆蓋熱圖。適合入門診斷或小型環境快速評估。
WiFi Analyzer
手機工具（Android）
即時顯示附近所有 AP 的 SSID、頻道、RSSI，視覺化頻道使用情況。適合快速現場評估，免費且易於使用。
ping / traceroute
基本診斷
確認基本連通性（ping 閘道器確認 Layer 3），識別延遲節點（traceroute），是所有排障的起始點。

—本章重點整理

Chapter 07 Summary
  • WMM（802.11e）將無線頻道分為四個優先佇列：AC_VO（語音，最高）、AC_VI（視訊）、AC_BE（一般，預設）、AC_BK（背景，最低）。EDCA 透過 AIFSN、CWmin/CWmax、TXOP 三個參數控制各佇列的競爭優先級，數值越小優先級越高。
  • DSCP EF（46）對映 AC_VO；AF41（34）對映 AC_VI；BE（0）對映 AC_BE；CS1（8）對映 AC_BK。訪客 SSID 必須設定不信任客戶端 DSCP，強制重標記為 AC_BE，防止濫用語音佇列。
  • 標準 WPA2 漫遊需 150～600 ms，遠超 VoIP 可接受的 50 ms 上限；802.1X 模式因需重新 RADIUS 認證延遲尤為嚴重。
  • 802.11k（鄰居報告）縮短掃描時間 60～80%；802.11r（FT 金鑰快取）將漫遊時間壓縮至 <50 ms；802.11v（BSS Transition）解決 Sticky Client 問題。三者合稱「漫遊三劍客」，應同時啟用。
  • 802.11r 有 Over-the-Air（<20 ms）和 Over-the-DS（<50 ms）兩種模式；少數舊裝置不相容，建議使用 Adaptive FT 模式。
  • Airtime Fairness 確保各裝置獲得相同空氣時間，防止低速舊裝置佔據大量頻道時間拖累高速裝置。在混合速率環境中效益顯著。
  • Rate Adaptation 根據 SNR 動態選擇最佳 MCS 索引；SNR >30 dB 可用最高 MCS，SNR <10 dB 僅能使用 MCS 0（BPSK）。
  • 效能排查順序：有線端確認 → RSSI/MCS → 頻道使用率/重傳率 → AP 負載 → 低速裝置干擾。
  • VoIP 品質以 MOS Score 衡量；延遲 <100 ms、抖動 <20 ms、遺失 <1% 可達 MOS ≥ 3.5。斷音多為漫遊問題，機器音多為抖動過高問題。
  • 故障排除應從 OSI Layer 1 開始逐層排查，不應跳層猜測。常用工具：iPerf3（速率量測）、Wireshark（封包分析）、Ekahau（Site Survey）。
